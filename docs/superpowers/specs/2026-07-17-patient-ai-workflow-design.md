# 护无忧患者资料与 AI 陪诊流程改版设计

## 目标

在现有纯前端演示项目内，形成患者需求提交、资料上传、医院申请、价格展示、后台处理、通知反馈和陪诊报告查看的完整闭环。手动表单与 AI 助手使用同一份需求草稿，患者可以随时切换，不重复填写。

本期是可演示原型：数据保存在浏览器本地，身份证 OCR 只提供明确标注的演示识别结果。代码预留真实 OCR、文件存储、AI 和消息服务接口，但不把模拟能力描述为生产能力。

本期仅覆盖同一浏览器中的单个演示患者与单个演示管理员切换使用。真实登录、多人并发、跨设备同步、服务端消息推送和医疗数据生产合规不在本期范围内。

## 患者端

### 需求提交

- “我的需求”提供“手动填写”和“AI 帮填”两个入口，共享同一份草稿。
- 两个入口地位相同，患者可随时切换；任何一侧修改都立即同步到另一侧，不重复填写。
- 必填信息为患者、医院、科室、日期和服务类型；特殊需求、身份证和检查报告可后补。
- 合作医院可直接选择。未收录医院通过“申请新医院”提交后台审批，不进入医院对接人流程。
- 服务类型实时显示管理员维护的公开价格，提交前展示费用确认卡。

### 资料与报告

- 身份证支持正反面拍照或上传、预览、替换和删除。
- 点击识别后展示姓名、身份证号、性别和出生日期等可编辑结果，并要求患者确认。
- 检查报告支持多图上传、预览和删除。仅接受 JPEG、PNG、WebP；原文件单张不超过 10MB，最多 6 张；保存前压缩为最长边 1600px、单张目标不超过 800KB。类型不符、超过数量或压缩失败时拒绝对应文件并说明原因，不清空已选图片。
- 已完成需求展示“陪诊报告”：服务时间线、完成事项、陪诊总结、报告图片和患者可理解的说明。
- 陪诊报告由管理员在需求完成后填写并发布，发布前患者只能看到服务进度，发布后可查看原始内容与通俗说明。
- 通俗说明在原型中由固定规则模板根据管理员填写的摘要生成，管理员可修改后再发布；不自动解读报告图片。它只做信息整理，不下诊断、不修改医生结论，并提示以医生意见为准。

### 医院与价格

- 患者端展示服务收费清单，包括名称、服务范围、价格和计价单位。
- 医院列表展示医院名称、等级、地址、优势科室和简短介绍。
- 后台修改价格或医院介绍后，患者端未提交页面立即读取更新数据。需求提交时保存服务名称、价格和单位快照，后续改价不影响历史需求。

## AI 助手

AI 使用显式状态机，而不是仅返回自由文本：

1. 识别用户意图是新建需求、查询进度、了解医院、查询价格或补充资料。
2. 新建需求时逐项抽取医院、科室、日期、服务类型和特殊需求。
3. 信息不完整时只追问当前最关键的一项，避免一次要求患者填写整张表。
4. 基础信息齐全后展示结构化确认卡，患者可确认或切换到手动表单修改。
5. 确认后依次展示身份证上传卡、检查报告上传卡和最终提交卡。
6. AI 产生的字段实时写入共享草稿；手动表单修改也同步回 AI 上下文。
7. 最终提交卡明确展示服务内容、公开价格、资料用途和下一步，患者主动确认后才创建需求。

助手需要明确能力边界：不提供诊断和用药决定；身份证图片和病历属于敏感数据，上传前提示用途；提交前必须由患者确认。

演示环境默认使用本地规则与状态机完成字段抽取，保证离线也能走通。`AiAssistantService.interpret({ text, draft, intent })` 只返回 `{ intent, fields, missingFields, reply }`，其中 `fields` 只能包含草稿允许字段，不能创建需求或改变状态。界面状态机负责合并草稿、决定下一项追问、渲染确认/上传/提交动作卡以及最终提交。未来接入真实大模型时仅替换该解释服务，流程状态和提交权限不变。

AI 会话阶段固定为 `idle → collecting → confirm_details → upload_identity → upload_reports → final_confirm → submitted`。患者切换到人工表单不会重置阶段；表单字段变更后重新计算缺失项。返回上一步允许修改，只有 `final_confirm` 的明确确认操作可以创建需求。

## 管理员端

- 删除收入、账单、抽佣、待结算、已结算等财务结算数据和界面。
- 保留“收费标准”页面，仅用于服务项目价格维护和变更记录。
- 删除需求详情中的医院对接人、确认对接和绿色通道操作。
- “医院审批”只处理患者提出的新医院申请，可通过或填写原因后驳回；通过后进入医院资料库。
- 管理员可编辑医院名称、等级、地址、优势科室和患者端简介。
- 需求详情展示身份证识别状态、检查报告和陪诊报告；敏感号码默认脱敏。
- 已完成需求提供陪诊报告编辑器，可填写服务时间线、已办事项、陪诊总结、就医注意事项并上传报告图片，确认后发布给患者。
- 管理员接收新需求和新医院申请提醒；患者接收需求已分配、服务已开始、服务已完成和陪诊报告已发布提醒。通知可跳转到对应记录并支持标记已读。

## 数据与接口边界

- 使用统一 Store 管理需求、草稿、价格、医院、医院申请、通知和陪诊报告，并写入 `localStorage`。
- 图片压缩后以 Data URL 保存，仅用于原型；达到浏览器容量限制时给出明确提示。
- OCR 通过 `IdentityOcrService.recognize({ file, side })` 调用，`side` 为 `front` 或 `back`。返回 `{ success, side, fields, confidence, error }`；正面字段为姓名、身份证号、性别、出生日期，反面字段为签发机关和有效期限。无法识别时保留图片并允许重试或手工录入；任何识别字段都可编辑，患者确认后记录 `identityStatus: confirmed`。重传任一面会把状态退回 `unconfirmed`。原型实现返回明确标注的演示数据，生产实现替换为后端 OCR。
- AI 通过 `AiAssistantService` 调用。演示环境不内置第三方 API 密钥；生产环境必须由后端代理，密钥不能硬编码或保存在浏览器中。
- 所有用户输入在插入 HTML 前转义，文件校验类型和大小。
- AI、OCR 或图片持久化失败时展示可操作的错误提示，并保留已填写草稿；人工表单始终可继续使用。

### 核心数据模型

- `Draft`：患者、联系方式、医院/医院申请、科室、日期、服务类型、特殊需求、身份证图片与确认字段、检查报告图片。固定键 `current-patient-draft`。
- `Need`：唯一 `id`、患者快照、就诊字段、`serviceSnapshot`、资料引用、状态、创建/更新时间、陪诊师、可选 `escortReportId`。
- `PriceItem`：唯一 `id`、名称、说明、价格、单位、启用状态；`PriceChange` 记录项目、原价、新价和时间，仅在收费页面展示最近变更。
- `Hospital`：唯一 `id`、名称、等级、地址、优势科室、患者端简介和启用状态。
- `HospitalApplication`：唯一 `id`、患者、医院名称、科室、原因、状态、驳回原因、关联需求草稿和处理时间。
- `Notification`：唯一 `id`、接收角色、类型、标题、时间、已读状态，以及 `{ targetType, targetId }` 跳转目标。
- `EscortReport`：唯一 `id`、需求 `needId`、时间线、已办事项、陪诊总结、注意事项、报告图片、通俗说明、草稿/已发布状态和发布时间。

Store 使用单一版本化键 `huwuyou_store_v1`。加载时合并默认字段；JSON 损坏时备份原字符串到 `huwuyou_store_corrupt_backup`、恢复演示初始数据并提示用户。所有写入先完成序列化，失败时保持内存旧值并提示存储空间不足。实体之间只通过唯一 ID 关联，删除目标后跳转页显示“记录不存在”并返回列表。

### 需求生命周期

允许状态转换为：`待处理 → 已分配 → 服务中 → 已完成`，以及 `待处理/已分配 → 已取消`。分配陪诊师触发“已分配”患者通知；管理员点击“开始服务”触发“服务中”；点击“完成服务”触发“已完成”；只有已完成需求可以编辑和发布陪诊报告。发布报告不改变需求状态，只写入 `escortReportId` 并产生“陪诊报告已发布”患者通知。禁止跳级操作，界面隐藏不允许的按钮，Store 再次校验。

### 医院申请闭环

未收录医院可以先保存到需求草稿，但相关需求在医院获批前不能最终提交。相同患者、相同医院名称存在待审批申请时复用原申请并提示“审核中”。通过后创建或关联医院记录、回填草稿并通知患者；驳回必须填写原因，患者可修改医院名称或原因后重新提交为新申请。审批记录保留，不覆盖历史结果。

### 通知规则

演示身份由当前患者端/管理员端路由决定，双方只读取 `recipientRole` 匹配的通知。通知事件用 `type + targetId + 状态` 作为去重键，同一次状态写入不会重复创建。点击通知先标记已读，再按 `targetType` 跳转到需求或医院申请；目标不存在时显示提示并回到对应列表。未读状态随 Store 持久化，刷新后保留。

## 响应式设计

- 手机端移除固定手机壳感，使用 `width: 100%`、`min-height: 100dvh` 和安全区内边距。
- 桌面患者端使用固定上限宽度居中展示，但高度不随 Tab 内容变化。
- 内容区独立滚动，顶部栏、底部导航和 AI 输入区保持稳定。
- AI 面板在手机端使用底部全宽抽屉并处理软键盘，在桌面端使用受限宽高的浮层。
- 管理后台在窄屏切换为可收起导航和横向可滚动表格，保证网页端可操作。
- 重点验证 `375x812`、`430x932` 和常见桌面视口；切换 Tab、打开软键盘或展开 AI 面板时不改变应用外框尺寸。
- 可重复验收范围为最新版桌面 Chrome 的响应式模式，以及一台可用的 iOS Safari 或 Android Chrome 真机。若没有真机，软键盘仅列为人工待验项，不以桌面模拟结果宣称通过。

## 验收标准

- 患者能通过手动表单和 AI 两条路径创建同结构需求，切换后内容不丢失。
- AI 在收集完基础信息后按顺序显示确认、身份证、检查报告和提交操作。
- 身份证和检查报告可上传、预览、删除，刷新后原型数据仍存在。
- 完成需求可查看陪诊报告和通俗说明。
- 患者能查看收费和医院简介；管理员改价、改简介后患者端同步更新。
- 后台不存在财务结算和医院对接人流程，只保留收费标准与医院审批。
- 新需求和关键状态变化会生成可跳转通知。
- 375px 手机视口、430px 手机视口和桌面视口下布局尺寸稳定，无横向溢出。

## 验证

- 对 JavaScript 执行语法检查。
- 使用浏览器分别验证患者手动提交、AI 提交、资料上传、医院申请、后台审批、改价、改医院简介和通知跳转。
- 在 375x812、430x932 和桌面视口检查布局、滚动、软键盘附近输入区和 Tab 切换尺寸。
